# Documentation Guardrails

This file defines how a local LLM or review assistant should evaluate the product documentation.

The guard must not invent product decisions or silently rewrite requirements.  
Its role is to check structure, classification, traceability, consistency, and requirement quality.

---

## 1. Guard Role

The guard acts as a documentation reviewer.

It may:

- Classify new notes
- Suggest where content belongs
- Identify missing traceability
- Detect duplicate or overlapping requirements
- Flag vague or untestable requirements
- Identify orphan requirements
- Identify possible scope creep
- Identify MVP inconsistencies
- Suggest questions to add to Open Questions
- Suggest decisions to add to Decision Log

It must not:

- Invent new product capabilities
- Make final product decisions
- Rewrite the product direction
- Add implementation details without being asked
- Treat suggestions as accepted requirements

---

## 2. Review Categories

The guard should review documentation using the following categories:

### 2.1 Classification

Check whether each item belongs in the correct section:

- Product Overview
- Users and Roles
- Product Requirements
- User Requirements
- System State
- System Requirements
- Non-Functional Requirements
- Architecture
- Block-Level Specification
- MVP
- Roadmap
- Parking Lot
- Decision Log
- Open Questions

### 2.2 Traceability

Check whether requirements are linked correctly:

Product Requirement → User Requirement → System Requirement → Block-Level Requirement → MVP/Roadmap

### 2.3 Requirement Quality

Check whether requirements are:

- Clear
- Specific
- Testable
- Not duplicated
- Not overly broad
- Not too implementation-specific too early
- Assigned to the correct level

### 2.4 Consistency

Check whether sections contradict each other.

### 2.5 Scope Control

Check whether items belong in MVP, roadmap, parking lot, or current requirements.

### 2.6 Open Questions

Identify unresolved issues that should be added to `docs/3.open_questions.md`.

### 2.7 Decisions

Identify important accepted choices that should be added to `docs/2.decision_log.md`.
